# Duplicate allocations audit — 2026-04-25

Investigation-only. No code, schema, or data changes.

## Scale

- **32 duplicate `(plan_id, asset_class)` groups** in `pension_allocations`.
- **92 of 190 rows involved** (48%).
- Spread across the thorough-tier plans plus a small TRS Texas tail.

## Root cause — three patterns

### Pattern 1 — Sub-class breakdowns (~70% of duplicates, legitimate)

The classifier correctly emits one row per policy sub-sleeve when a plan's investment policy splits an asset class into multiple separately-targeted sleeves. v1.1-cafr was built for this (`lib/classifier/prompts/cafr-allocation.ts`).

Canonical examples:

- Michigan SMRS Public Equity → `Domestic` + `International` rows.
- NYSTRS Fixed Income → `Domestic FI` + `High-Yield` + `Global Bonds`.
- NYSTRS Credit → `Real Estate Debt` + `Private Debt`.
- TRS Texas Public Equity → `USA` + `Non-US Developed` + `Emerging Markets`.

These are not data bugs. The `sub_class` column carries the verbatim sleeve label, and the unique index is `(plan_id, as_of_date, asset_class, coalesce(sub_class, ''))`, so the rows are correctly distinct in the data layer.

The duplicate count is generated by a query that groups on `(plan_id, asset_class)` only. Once `sub_class` is added to the grouping, these collapse.

### Pattern 2 — Time-series snapshots (~25%, legitimate)

Several plans have multiple `as_of_date` values for the same asset class because successive fiscal-year CAFRs were ingested.

- CalSTRS: 3 years of FY snapshots.
- A handful of others have 2 FYs each.

Not a data bug — the historical series is a feature, not a duplicate. Becomes a UI problem only when summary surfaces don't filter to the most recent `as_of_date` per plan.

### Pattern 3 — TRS Texas mixed-quality entries (2 rows, minor)

Two TRS Texas rows share an `(asset_class)` but were extracted from different source documents at different `as_of_date` values, producing mixed confidence tiers in the same group. Same root cause as Pattern 2 (no `as_of_date` filter on summary query) but localized to TRS Texas because of how its quarterly fact sheet vs. annual report are both present.

## Data integrity assessment

**The data layer is correct.** The unique index allows these rows by design (sub-sleeves get `sub_class`; FY snapshots get distinct `as_of_date`). What looks like duplication is a UI display issue — summary surfaces aggregate or pivot on `(plan_id, asset_class)` without `sub_class` or `as_of_date`, so legitimately distinct rows collapse into a misleading count.

No row deletion is warranted. No backfill is warranted.

## Recommended fixes (UI-only)

| # | Fix | Impact | Effort |
|---|---|---|---|
| 1 | Display `sub_class` in pension-profile allocation table — render "Public Equity · Domestic" / "Public Equity · International" as separate rows with the sleeve label as a secondary line | High — collapses ~70% of perceived duplicates immediately | ~1 hr |
| 2 | Filter to the latest `as_of_date` per `(plan_id, asset_class, sub_class)` in main views (pension profile, landing-hero composition, /signals allocation tab) | High — eliminates Pattern 2 + 3 confusion | ~1 hr |
| 3 | For aggregate summary cards (unfunded-budget hero, /plans list), aggregate top-level rollups: sum sub-classes back up to the parent asset class for headline math, but keep sub-class detail on the profile drilldown | Medium — keeps headline math intuitive while preserving sleeve granularity downstream | ~1 hr |

All three are UI-only. Total ≈ 3 hours.
