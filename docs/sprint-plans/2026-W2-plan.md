## Day-by-day breakdown

### Day 1 (2026-05-01) - Pipeline integration of verifyCrossSource

**Hours: 5-6**

Goal: cross-source verification runs automatically on new allocation arrival.

Tasks:
1. **Design call: where does the integration hook live?** (30 min)
   - Option A: Database trigger that fires a Vercel function on insert into `pension_allocations`. Cleanest separation, but introduces async coupling.
   - Option B: Inline call inside the CAFR / IPS classifier pipeline after a successful allocation insert. Simpler, but couples the verifier to ingestion latency.
   - Option C: Periodic batch job (every N hours) that picks up unverified allocation rows and runs the verifier. Easiest to reason about, slowest to surface results.
   - Recommendation: Option B for v1. Promote to A or C if latency or reliability suffers.

2. **Implement integration hook** (2 hours)
   - Add helper `verifyAllocationOnInsert(supabase, newAllocationRow)` in `lib/predictive/verify-cross-source.ts` or a new `lib/predictive/pipeline.ts`.
   - Helper finds candidate IPS rows for the same plan + asset class + sub_class via `buildVerifiablePairs`, runs `verifyCrossSource` on each, persists results.
   - Wire into the CAFR allocation classifier's per-row insert point and the IPS classifier's per-row insert point.

3. **Confidence-multiplier update on signals** (1 hour)
   - When a `source_verifications` row is written with `confirms` / `partially_confirms` / `policy_changed`, find related signal rows for the same plan + asset_class and update a `confidence_multiplier` field (NEW column, design call).
   - Multiplier values per Section 5e in build spec: 1.0 single-source, 1.5 two-source, 2.0 three+. Implementation question: do we materialize the multiplier on the signal row, or compute it on read via a view?

4. **Smoke test on existing data** (1 hour)
   - Re-run CAFR scraper for one plan. Confirm verifier auto-runs, writes rows, updates multipliers. Confirm idempotency (re-running doesn't double-count).

5. **Smoke test on new allocation flow** (30 min)
   - Manually insert a test allocation. Confirm trigger fires. Confirm rollback if any step fails.

Pre-flight verification:
- Read `lib/classifier/extract.ts` to understand the existing allocation-insert path.
- Confirm the ingestion path is idempotent (a re-run on the same doc produces no duplicates).

Risks / open questions:
- Where does the multiplier live? On `signals.confidence_multiplier` (new column) or computed in a `signals_with_verification` view? Open design call.
- Cost cap: each new allocation triggers up to ~5 model calls (one per IPS sub-sleeve match). At 100 allocations/day across all plans, ~500 model calls/day at ~$0.0035 each = ~$1.75/day. Acceptable.
- What happens when verification fails (network error, model timeout)? Currently `verifyCrossSource` throws. Pipeline should catch and continue ingestion, log the failure to a `verifier_failures` table or `error_message`.

### Day 2 (2026-05-02) - CAFR extraction quality fixes

**Hours: 4-5**

Goal: 2 real conflicts surfaced by Week 1 v1.1 verifier are fixed at the CAFR extraction layer.

Tasks:
1. **Investigate CalPERS Credit CAFR extraction** (1 hour)
   - Pull the CalPERS CAFR document(s) that produced the Credit allocation row at 3.5%.
   - Trace the extraction: did the classifier read a sub-sleeve number (Private Debt) and label it as the Credit parent? Or is the CAFR itself ambiguous?
   - If the issue is in the prompt: revise `lib/classifier/prompts/cafr-allocation.ts` to explicitly handle the Credit / Private Debt parent-vs-sub-sleeve case.
   - If the issue is the CAFR ambiguity: add CalPERS-specific calibration notes in the prompt.

2. **Re-extract CalPERS CAFR** (30 min)
   - Run the updated classifier against CalPERS CAFR. Confirm Credit row updates to the correct parent-class value.
   - Run `verifyCrossSource` on the new pair. Confirm verdict moves from `conflicts` to `confirms` or `policy_changed`.

3. **Investigate CalPERS Public Equity sub-sleeve issue** (1 hour)
   - The CAFR captures Public Equity at 37% (parent class). The IPS splits into Cap Weighted 27% + Factor Weighted 10% (sum = 37%).
   - Decide: should CAFR extract sub-sleeves to match IPS structure, or should the verifier prompt better handle parent-vs-sub-sleeve at known-roll-up cases?
   - If CAFR extraction: revise the prompt to extract Public Equity sub-sleeves where the source mentions them. This is actually orthogonal to the conflict; the v1.1 verifier mishandled the SAME-LEVEL pair, not a roll-up case.
   - If verifier prompt: bump to v1.2-allocation with explicit "X Weighted" / "X Active" / "Smart Beta" sub-sleeve guidance.
   - Recommendation: verifier prompt fix is cheaper; CAFR sub-sleeve extraction is a longer-tail design call. Do the prompt fix in Week 2; defer CAFR extraction redesign to a later week.

4. **Re-run verifier on the 29 pairs** (30 min)
   - With v1.2-allocation prompt, confirm Cap Weighted pair flips to `partially_confirms`.
   - Update the verifier_version in source_verifications rows.

5. **Update Day 6 findings doc with the resolution** (30 min)

Pre-flight verification:
- Read the CalPERS CAFR text to understand the source structure before changing the prompt.
- Re-read `lib/classifier/prompts/cafr-allocation.ts` and `lib/classifier/prompts/cafr-allocation-overrides.ts` (if exists) to understand existing plan-specific calibrations.

Risks / open questions:
- Does the CalPERS CAFR even surface the Credit parent at the right number? If the document itself shows 3.5% as the Credit-parent target, the issue isn't the classifier. Worth checking before assuming.
- The Public Equity sub-sleeve fix is a verifier-prompt issue, not a CAFR-extraction issue. Don't conflate them.

## Risks and mitigations

**Risk 1: Pipeline integration bug corrupts ingestion.**

Mitigation: Day 1 includes idempotency tests. The integration is wrapped in try/catch so a verifier failure doesn't roll back the parent ingestion. Worst case is a missing source_verifications row, which is recoverable via batch backfill.

**Risk 2: Mass PRIM IPS date-sweep finds nothing.**

Mitigation: The date-sweep pattern has fallbacks (try multiple months, try Jan/Apr/Jul/Oct first). If genuinely nothing matches, document the deferral and note the next IPS publication is the trigger to revisit.

**Risk 3: NYSCRF AJAX requires Playwright.**

Mitigation: Decision criterion documented. If the endpoint needs a session, defer to Week 3. Playwright introduces deployment complexity (browser install on Vercel runtime); not blocking Week 2.

**Risk 4: CAFR extraction fix doesn't actually fix CalPERS Credit.**

Mitigation: Day 2 starts with investigation, not implementation. If the source CAFR shows 3.5% genuinely, the issue is the source document, not our extraction. In that case, document and move on.

**Risk 5: plan_stakeholders schema design wrong on first try.**

Mitigation: The schema in spec Section 6 is a draft. Day 5 includes 30 min for a schema review against the actual scraper output before applying.
