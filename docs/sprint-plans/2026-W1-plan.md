## Streams to add this week

### Stream 1: Press releases

**What it captures:** Commitment announcements, consultant changes, RFP issuances - often disclosed via press release before showing up in official board minutes (sometimes by 30-60 days).

**Update frequency:** Daily check, only ingests new releases.

**Document type:** New `document_type` value: `press_release`.

**Classifier:** Reuses existing signal classifier; press release content goes through same Type 1/2/3 extraction. No new prompt needed in Week 1 - we test reuse first, then iterate if needed.

### Stream 2: Investment Policy Statement (IPS)

**What it captures:** Target allocations per asset class, pacing parameters, manager selection criteria, prohibited investments, geographic constraints.

**Update frequency:** Weekly check (IPS rarely changes; daily is overkill).

**Document type:** New `document_type` value: `ips`.

**Classifier:** New IPS classifier prompt needed. Extracts:
- Target allocation % per asset class
- Pacing constraints (annual deployment range)
- Manager criteria (size, geography, strategy)
- Diff against last version (signal: target allocation changed)

### Cross-source verification primitive

**What it does:** Takes 2 candidate signals from different streams, decides if they're the same event.

**Example use case:** A press release says "Board approved $500M to KKR Infra V" and a board minutes document a month later confirms it. The function matches them, marks them as cross-verified, increases the signal's confidence weight.

**Implementation:** Single function `verifyCrossSource(signalA, signalB)` returning `{ same_event: bool, confidence: 0..1, rationale: string }`. Uses Claude API with a focused prompt.

## Risks and mitigations

**Risk 1: Press release pages have anti-bot protection**

Mitigation: 4 of 5 plans (CalPERS, NYSCRF, Mass PRIM, Oregon PERS) have publicly accessible press release pages per existing scraper docs. CalSTRS may need Chrome UA spoofing per the research doc. If hard blockers emerge mid-week, swap that plan for LACERA or Virginia RS.

**Risk 2: IPS documents may not be at predictable URLs**

Mitigation: Day 1 includes URL verification. If a plan publishes IPS only as "search the documents page" rather than at a static URL, we either (a) implement a "find IPS in documents page" scraper, or (b) defer that plan's IPS to Week 2.

**Risk 3: Cross-source verification may not find matches in current data**

Mitigation: Current data has only CAFR + board minutes for most plans. Cross-source verification is most useful when we have both press releases AND board minutes. Week 1's verification testing may surface only theoretical capability rather than real matches. That's acceptable - we're proving the primitive works, not validating the data.

**Risk 4: Week 1 timeline slips**

Mitigation: If Day 4 doesn't complete press releases for all 5 plans, we still have Days 5-7 for IPS and verification. Critical-path is press releases for at least 3 plans + IPS for at least 3 plans + verification primitive. Reduced scope is acceptable.
